2 任务状态机

每个 Run 创建一个 ScheduledTask，经历以下状态流转。队列基于 SQLite 持久化，重启后可自动恢复。

QUEUED 入队等待，按优先级 + 创建时间排序
→
DISPATCHED Worker 认领，绑定 worker_id
→
RUNNING 子进程已启动，正在流式输出事件
→
COMPLETED 流程正常完成，触发 Hooks
↻ 失败时：RetryEvaluator 判断 → 重新入队 (QUEUED)
✕ 不可重试 → FAILED
⊘ 用户取消 → CANCELLED

3 请求完整流转

一次 Run 从 API 提交到最终完成的端到端流程。

API POST /runs
验证参数 → 创建 Run (status=QUEUED) → 构建 RunConfigHelper.build_v1() → 发送 SSE 事件
Dispatch RunDispatcher.dispatch()
通过 SchedulerDispatcher 将 run_id 传递给调度器（替换旧的 in-memory TaskRunner）
Scheduler 入队 & Worker 认领
TaskQueue.enqueue() 创建 ScheduledTask → Worker 循环 claim_next() 按优先级 (URGENT > NORMAL > LOW) + FIFO 取出 → 并发受 max_concurrency 信号量控制
Resource 资源检查 (可选)
ResourceMonitor 采样 CPU/内存/磁盘/GPU → ResourceEstimator 根据配置或管线名称预估需求 → ResourceChecker 确认可用 ≥ 所需 + 安全余量 → 不足则重新入队延迟 30s
Engine 执行工作流
adapter.pre_submit() 准备环境 (Docker 检测、profile 注入等) → adapter.build_command() 构建 CLI 命令 → LocalBackend.submit() 通过 asyncio.create_subprocess_exec 启动子进程 → 流式读取 stdout/stderr，逐行调用 adapter.parse_event()
Events EngineEvent 事件流
每个事件实时处理： TASK_UPDATE → DagMatcher 模糊匹配更新 DAG 节点 | LOG → 追加日志 + SSE | ERROR → 记录错误信息 | COMPLETED → 最终化 DAG 状态
Retry 失败处理
RetryEvaluator.should_retry() 检查：尝试次数 < 上限？错误匹配模式 (connection/oom/137/killed)？→ 是则计算指数退避延迟 delay × multiplier^(attempt-1) → 重新入队
Hooks 完成后处理
RunCompletionHooks.on_run_terminal() → ① 工作目录清理 (CleanupPolicy) → ② 审计日志 (AuditService) → ③ Webhook 通知 (NotificationService) → ④ 批量状态更新 (BatchService)

4 核心组件

各组件职责与关键方法一览。

TaskQueue

scheduler/queue.py

DB-backed 优先级队列，SQLite 持久化。使用 asyncio.Lock 序列化并发操作。

  • enqueue() 创建任务
  • claim_next(worker_id) 原子认领
  • re_enqueue() 重试重新入队
  • get_stale() 发现僵尸任务

RunScheduler

scheduler/scheduler.py

调度器核心，管理 Worker 池、资源检查、超时监控。启动时自动恢复 stale 任务。

  • start() 启动 Workers + 监控
  • enqueue() 提交新任务
  • cancel() 取消排队/运行中任务
  • recover() 重启后恢复

LocalBackend

engine/local.py

本地子进程执行器。通过 asyncio 异步 spawn 进程并流式读取输出。

  • submit() 启动进程 → yield 事件流
  • cancel() SIGTERM → SIGKILL
  • 双通道 drain (stdout + stderr)

EngineAdapter

engine/adapter.py

引擎抽象接口 (ABC)。Nextflow 和 WDL 各自实现，通过 EngineRegistry 自动注册。

  • build_command() 构建 CLI 参数
  • parse_event(line) 解析输出行
  • pre_submit() 预处理 (Docker 等)
  • extract_schema() 提取 DAG Schema

RetryEvaluator

scheduler/retry.py

指数退避重试策略。预设生信场景匹配模式 (connection, oom, 137, killed)。

  • should_retry() 判断是否重试
  • next_delay() 计算退避延迟
  • 默认策略：2 次重试，60s 起始

ResourceMonitor

scheduler/monitor.py

系统资源采样器。使用 psutil 定期采集，检测 GPU 通过 nvidia-smi。

  • 采样：CPU / 内存 / 磁盘 / GPU
  • ResourceEstimator：按管线名预估
  • ResourceChecker：可用 ≥ 需求 + 余量

DagMatcher

utils/dag_matcher.py

模糊匹配运行时任务名到 Schema DAG 节点。4 级策略：精确 → 标签 → 后缀 → 子串。

  • Schema DAG：注册时静态解析
  • Runtime DAG：执行时动态构建
  • 无 Schema 时自动创建节点

BatchService

services/batch_service.py

批量提交管理。支持 2-500 个 Run 同时提交，跟踪整体完成状态。

  • create_batch() 批量创建 Runs
  • update_batch_status() 聚合状态
  • 状态：PARTIAL (部分成功/失败)
  • 终态时触发 on_batch_complete

8 关键设计决策

整个架构中反复出现的设计哲学。

SQLite 单实例

不引入 Redis / RabbitMQ / Celery。SQLite 作为持久化队列，Python async loop 驱动调度。足够简单，保留 PostgreSQL 迁移路径。

Best-Effort 哲学

Webhook 通知、WDL 恢复、资源预估均为"尽力而为"。不做强一致性保证，换取实现简单和系统韧性。

Feature Flag 回滚

run_scheduler_mode = "legacy" | "persistent"，资源检查可独立开关。每个阶段可安全回退。

延迟而非拒绝

资源不足时 re-enqueue + 30s delay，不是 reject。保证任务最终执行，避免用户困惑。

Hooks 扩展点

RunCompletionHooks 统一管理完成后副作用（清理、审计、通知、批量更新），避免逻辑散落各处。

不可变配置命名空间

RunConfigHelper 用 v1 schema 分离 request / resolved / runtime / policy / ui，防止字段冲突，运行时追加不污染原始配置。
