# Bioinfoflow — 产品概述

## 一句话

Bioinfoflow 是一个本地优先、AI 驱动的生物信息学平台，让研究人员通过自然语言编排复杂的分析流程——数据不离开你的机器，不被云平台锁定。

## 问题

生物信息学的基础设施困在两个糟糕的选择之间：

1. **云平台**（Terra、DNAnexus、Seven Bridges）：功能强大但昂贵，锁定你的数据，且迫使你将敏感的基因组数据上传到别人的服务器。合规性问题不断，供应商锁定严重。
2. **原生命令行工具**（手动运行 Nextflow/WDL）：完全可控，但学习曲线陡峭，没有项目管理，没有调度器，看不到什么在运行。每个实验室都在重新发明同样的包装脚本。

与此同时，研究人员把 30-60% 的时间花在基础设施上，而不是科学研究。他们想在自己的数据上运行 `nf-core/rnaseq`，而不是调试 Nextflow 配置文件。

## 解决方案

Bioinfoflow 运行在你的机器（或实验室的服务器）上，提供：

- **AI 智能体** —— 理解生物信息学工作流，你说"对这些 FASTQ 文件做差异表达分析"，它会找到合适的流程、参数并执行。
- **双引擎支持** —— 通过统一接口运行 Nextflow 和 WDL/MiniWDL 流程。你现有的工作流无需修改。
- **现代 Web UI** —— 项目管理、实时运行监控、DAG 可视化、文件浏览器、集成终端。不是又一个终端界面。
- **CLI 工具（`bif`）** —— 面向高级用户和自动化。完整的 API 覆盖，流式输出，NDJSON 格式便于脚本集成。
- **持久化调度器** —— 资源感知队列，支持重试、超时、清理和批量运行。不用再盯着长时间运行的流程。

一切都在本地运行。你的数据永远不会离开你的机器。SQLite 数据库，除了 LLM API 密钥外零外部依赖。

## 目标用户

- **生物信息学研究人员** —— 经常运行分析流程，厌倦了基础设施的开销。
- **计算生物学团队** —— 需要项目组织、审计追踪和运行状态的共享可见性。
- **核心设施/测序中心** —— 为多个实验室处理样本，批量运行、调度和资源管理至关重要。
- **拥有敏感基因组数据的团队** —— 无法或不愿将数据上传到云平台。

## 工作原理

```
用户（聊天或 CLI）
  → AI 智能体（理解生信上下文，调度工具）
    → 工作流引擎（Nextflow 或 MiniWDL 适配器）
      → 本地执行（带资源监控）
        → 实时结果（SSE 流式推送到 UI）
```

1. **描述你想做什么** —— 在聊天界面或通过 `bif agent chat`。
2. **智能体规划并执行** —— 读取你的文件，选择合适的工作流，配置参数，验证输入后提交。
3. **调度器排队并运行** —— 遵守资源限制（CPU、内存、磁盘、GPU），失败时重试，强制超时。
4. **实时查看进度** —— DAG 进度、日志、任务状态全部通过 SSE 流式推送到 UI。无需刷新。
5. **结果归档到项目** —— 输出、日志和审计记录按项目组织。

高风险操作（如删除运行或修改配置）需要通过审批工作流——智能体会在操作前征求你的同意。

## 为什么选择本地优先

这是一个刻意的架构选择，不是限制：

- **数据主权** —— 基因组数据是敏感的。HIPAA、GDPR、机构政策通常禁止上传到第三方云。本地优先意味着默认合规。
- **无供应商锁定** —— 你的流程是标准的 Nextflow/WDL。你的数据留在你的文件系统。随时可以离开。
- **成本** —— 实验室的 GPU 和机构 HPC 已经付过钱了。云计算账单对基因组学工作负载来说增长很快。
- **延迟** —— 大数据集无需上传/下载开销。在数据所在的地方运行。
- **离线可用** —— 除了 LLM API 调用，一切都可以在没有网络的情况下运行。

## 架构概览

- **后端**：Python 3.13+、FastAPI、async SQLAlchemy、SQLite（via aiosqlite）
- **前端**：Next.js 16、React 19、Radix UI、Tailwind CSS 4、React Flow
- **智能体**：Runtime v2 —— 显式异步循环，工具调度，上下文压缩，回合间钩子
- **引擎**：Nextflow 适配器（GPU 检测、断点续传）+ MiniWDL 适配器
- **调度器**：数据库支持的优先级队列，资源门控，重试，超时，清理，完成钩子
- **CLI**：Typer + Rich，3 种传输模式（remote/local/auto），流式支持
- **认证**：Better Auth（基于会话，目前仅前端强制）
- **实时**：SSE 广播运行事件、智能体事件、镜像进度

## 当前状态——诚实评估

**运作良好的部分：**
- 从自然语言到流程完成的完整生命周期
- 调度器处理并发运行，具备资源感知能力
- UI 切实好用——DAG 可视化、实时日志、文件浏览器、终端
- CLI 达到生产质量，具备完善的错误处理和流式输出

**尚未完成的部分：**
- 认证仅限前端——后端 API 是开放的
- 仅支持单节点——无分布式执行
- 无多用户协作功能
- 内置 starter workflow 市场已提供，社区共享尚未实现
- 测试覆盖率扎实但未覆盖所有边界场景

**我们现在不做的事：**
- 云执行后端——切入点是本地优先
- 流程编写——我们编排现有的 Nextflow/WDL，不是替代它们
- 电子实验室笔记本——专注自己的领域

## 下一步

近期优先事项，基于用户反馈：

3. **远程执行** —— 基于 SSH 的实验室服务器和机构 HPC 执行

5. **增强智能体能力** —— 更好的工作流推荐、参数调优、结果解读
