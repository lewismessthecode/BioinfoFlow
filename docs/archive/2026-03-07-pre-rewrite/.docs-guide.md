# Documentation Governance Guide

**Last Updated:** 2026-03-07

This guide defines how documentation is organized, maintained, and updated in the Bioinfoflow project.

## Document Hierarchy

| Status | Meaning | Documents |
|--------|---------|-----------|
| **CANONICAL** | Single source of truth. Must be kept current. | 01-product-overview, 02-architecture, 03-api-reference, 04-frontend-design, 05-implementation-plans, 06-development-log, CONTRIB, RUNBOOK |
| **REFERENCE** | Detailed specs for subsystems. Updated when the subsystem changes. | 10-agent-orchestrator-spec |
| **PLANNING** | Active sprint plans. Archived after completion. | Files in `plans/` |
| **ROADMAP** | Future version planning. Updated during roadmap reviews. | 14-v0.2-prd |
| **FUTURE VISION** | Long-term strategic directions not on the current roadmap. | 09-product-memo, 11-product-final-vision |

## Update Guidelines

### When to Update CANONICAL Docs
- After shipping a major feature or architectural change
- Monthly review (first week of each month)
- When a new contributor asks a question not covered by existing docs

### How to Create Planning Docs
- Use date-based naming: `YYYY-MM-DD-<topic>.md`
- Place in `docs/plans/`
- Include: goal, scope, implementation steps, verification criteria

### When to Archive Plans
- After all tasks in the plan are completed
- Move to `docs/plans/archive/`
- Update the plan table in `05-implementation-plans.md`

## Quality Standards

1. All documents must have a "Last Updated" date in the header
2. All documents must have a status tag in the title: `[CANONICAL]`, `[REFERENCE]`, `[FUTURE VISION]`, etc.
3. Cross-references should use relative links (e.g., `./02-architecture.md`)
4. Avoid duplicating content across documents; link to the canonical source instead
5. Keep the `docs/README.md` index up to date when adding or removing documents
