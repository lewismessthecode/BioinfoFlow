# Bioinfoflow Agent Orchestrator SPEC (Unified) [REFERENCE]

## Executive Summary

The Agent Orchestrator is the core subsystem that governs how Bioinfoflow's AI agent plans, executes, and audits bioinformatics workflows. It replaces the initial LangGraph loop with a **deterministic kernel** architecture where the LLM handles decision-making and planning, while a structured executor controls all side effects (file writes, run creation, image pulls).

**Key components:**
- **Deterministic Kernel** (`orchestrator/`): State machine that controls execution flow, enforces policies, and manages approvals. The LLM produces structured `ExecPlan` documents; the kernel executes them step-by-step.
- **Tool Runtime**: Unified interface for all agent tools (workspace, run, workflow, image operations) with risk-level classification (`read`, `act_low`, `act_high`), schema validation, and audit logging.
- **Evidence Chain**: Every agent run produces a file-based audit trail (`plan.md`, `exec_plan.json`, `action_log.jsonl`, `provenance.json`) stored in the workspace under `.bioinfoflow/runs/<run_id>/`.

**Implementation status:** The current system uses a LangGraph tool-calling loop with basic planning. The orchestrator spec defines the target architecture for P0, with migration via feature flags. Core tool runtime and risk classification are partially implemented. The approval system and full evidence chain are planned.

> 目标：让 Bioinfoflow 后端 agent 的”可用性”达到 Codex / Claude Code 的体验：可控、可复现、可审计、可渐进自动化，而不是”看起来很聪明但经常跑偏”的黑盒循环。

## 0. 这份 SPEC 取代了什么

本 SPEC 是把以下几类想法重新组合后的**唯一**版本：
- “Claude Code 级别工具集 + 多步规划/执行”思路（文件/搜索/执行/验证/通知）。
- “Agent run/step/toolcall/approval 数据模型 + 可视化时间线 + 审批卡片”思路（更像 IDE 里的 code agent）。
- “Deterministic Orchestrator Kernel + run 目录证据链 + workflow/image/run 一等公民 + 资产生命周期”思路（更像可运营系统）。

同时遵循：
- 第一性原理：LLM 不可靠，必须让**确定性内核**控制副作用和状态；可用性来自“约束 + 记录 + 恢复”，不是更复杂的 prompt。
- 奥卡姆剃刀：能用文件系统当证据链就别引入两套存储；能用少量状态机就别引入多层图编排；能用少量 Golden Tasks 验收就别定义一堆“未来可能需要”。

## 1. 问题定义（为什么现在不好用）

现状痛点（以 Codex / Claude Code 的使用体验为对照）：
- **不可控**：agent 何时行动、做了什么、为什么做，缺乏稳定的状态机与审批准入。
- **不可复现**：一次对话的运行资产（workflow 快照、inputs、params、镜像、命令、日志）缺少统一归档，难以复盘/再跑。
- **不可观察**：工具调用、耗时、输出摘要、失败原因、重试策略缺少结构化记录与 UI 呈现。
- **不易修复**：失败后缺少“分类->修复->重试/恢复”的确定性循环；也缺少“从任意 step 继续”的支点。

## 2. 设计目标与非目标

### 2.1 P0 设计目标（必须）

1) **Deterministic Kernel**：LLM 只负责“决策/规划”，内核负责“权限/执行/状态/回放”。  
2) **Human-in-the-loop**：任何高风险动作都可被策略拦截并走审批流（diff/命令/发布/运行）。  
3) **证据链**：每个 agent run 必须落盘可审计（plan、exec_plan、action_log、provenance）。  
4) **Run/Workflow/Image 一等公民**：agent 通过服务层工具操控它们，而不是在 prompt 里“假装做了”。  
5) **最小兼容**：保持现有 `/api/v1/agent/message` + SSE 主事件不崩（前端不升级也能工作）。

### 2.2 P0 非目标（明确不做）

- HPC/SLURM/K8s 等执行后端。
- 默认全自动的生产环境（prod 默认 guarded）。
- 复杂“workflow 稳定版回归矩阵+自动发版”完全落地（先做 beta 可用闭环）。
- 模型微调、RAG 大工程（先做 facts cache + on-demand refresh）。

## 3. P0 Golden Tasks（验收用例）

P0 的“好用”用一组任务定义（通过则算 V1 可用）：

- **GT1** Nextflow：创建 run -> 启动 -> SSE 流式日志/状态 -> 列出 outputs。  
- **GT2** WDL/MiniWDL：生成 `inputs.json` -> 启动 -> 流式日志/状态 -> outputs。  
- **GT3** Workspace FASTQ 自动识别：paired/single 识别 -> 生成 samplesheet -> 选 workflow -> 创建 run（审批策略生效）。  
- **GT4** 缺失 workflow：生成最小可跑模板（NF + WDL 二选一或都支持）-> 修到可跑 -> 注册到 registry（beta）。  
- **GT5** 缺失 image：识别并 pull（allowlist + policy）。  
- **GT6** 一次自愈：失败分类 -> 提出修复 -> 应用修复（inputs/config/workflow）-> retry/resume。  
- **GT7** provenance：run 的 provenance manifest（workflow 版本/通道、inputs、params、images、命令、关键日志摘要、产物索引）。

## 4. 核心架构：Kernel + ToolRuntime + Ledger

### 4.1 内核原则：把“不确定性”围起来

- LLM 的输出只允许是**结构化计划**（ExecPlan），不得直接产生副作用。
- 所有副作用必须通过 ToolRuntime 执行（统一校验、权限、记录、事件）。
- Executor 是确定性状态机：同样的 state + plan + tool 结果 => 同样的状态迁移与记录。

### 4.2 关键模块（后端目录）

建议新增/重构目录（以 `backend/app/services/agent/` 为根）：

- `orchestrator/state.py`：可序列化 RunState（facts、约束、指针、统计）。
- `orchestrator/policy.py`：风险分级、环境模式、allowlist、硬禁规则。
- `orchestrator/planner.py`：LLM 生成 ExecPlan + plan.md（PlanDoc）。
- `orchestrator/executor.py`：执行循环、重试/恢复、审批等待。
- `orchestrator/tool_runtime.py`：统一调用工具（schema 校验/风险/记录/事件）。
- `orchestrator/ledger.py`：写 ActionLog、Provenance、Artifacts 索引。

> 说明：当前 LangGraph loop 不做“魔改”，而是逐步被 `orchestrator/*` 替换，并通过 feature flag 切换。

## 5. 数据与存储：DB 做索引，文件系统做证据链

### 5.1 Run 目录（证据链，source of truth）

统一归档到：

```
<workspace>/.bioinfoflow/runs/<run_id>/
  plan.md
  exec_plan.json
  action_log.jsonl
  provenance.json
  inputs/
    samplesheet.csv (optional)
    params.json
    inputs.json
    config_overrides.json
  workflow/
    (snapshot of workflow used by this run)
  logs/
    run.log (or engine specific)
  artifacts/
    (tables/plots/previews generated by agent tools)
```

约束：
- run 目录一旦写入，尽量**只追加**（append-only），避免覆盖历史证据。
- 允许生成“新版本 provenance/plan”（例如 `plan.v2.md`）但需保留旧版本。

### 5.2 DB（UI/查询用索引）

DB 只存“可查询的结构化索引”，大对象落盘：
- `agent_sessions`：与 conversation/project 的绑定 + policy mode。
- `agent_runs`：每次执行的 run 记录（状态、模型、timestamps、workspace、engine 选择）。
- `agent_plans`：plan 元信息（版本、摘要、exec_plan 路径/hash）。
- `agent_steps`：step 状态机（PENDING/RUNNING/SUCCEEDED/FAILED/WAITING_APPROVAL/CANCELLED）。
- `agent_tool_calls`：每次工具调用的摘要（tool、duration、status、output_preview、artifact refs）。
- `agent_approvals`：审批请求（type、payload、status、resolved_by）。
- `workflow_drafts`：草稿工作流（draft_id、base_workflow_id、path、diff summary、status）。

> DB 是 index；文件系统是证据链。避免“DB 一套、文件一套、对不上”的双写复杂度。

## 6. ExecPlan：计划的最小可执行表示

### 6.1 ExecPlan（JSON）最小结构

`exec_plan.json` 推荐形状（示意）：

```json
{
  "version": 1,
  "goal": "Run nf-core/rnaseq on detected FASTQ",
  "constraints": {
    "policy_mode": "SAFE_AUTO",
    "workspace": "demo/sars-cov-2",
    "engine_preference": ["nextflow", "miniwdl"]
  },
  "steps": [
    {
      "id": "facts.scan_workspace",
      "title": "Scan workspace and detect FASTQ",
      "tool": "workspace_scan",
      "args": {"path": "demo/sars-cov-2"},
      "risk": "read",
      "depends_on": [],
      "retry": {"max_attempts": 1}
    },
    {
      "id": "workflow.pick",
      "title": "Select a workflow",
      "tool": "workflow_resolve",
      "args": {"query": "sars-cov-2"},
      "risk": "read",
      "depends_on": ["facts.scan_workspace"],
      "retry": {"max_attempts": 1}
    },
    {
      "id": "run.create",
      "title": "Create and start a run",
      "tool": "run_create",
      "args": {"workflow_id": "…", "workspace": "…", "params": {}},
      "risk": "act_high",
      "depends_on": ["workflow.pick"],
      "approval": {"required": true, "type": "run"}
    }
  ]
}
```

原则：
- LLM 可以更聪明，但 ExecPlan 必须“蠢而可跑”：一组工具调用 + 依赖 + 风险等级 + 重试策略。
- PlanDoc（`plan.md`）用于给人读；ExecPlan 用于给机器跑。

## 7. ToolRuntime：把工具统一成可审计的系统调用

### 7.1 工具分层（P0）

P0 工具集（只保留“必须用”的）：

**Workspace & 文件**
- `workspace_scan`：识别 FASTQ/BAM/VCF 等，并产出结构化样本表。
- `file_read`, `file_write`, `file_edit`：文件读写与精确编辑（带路径策略）。
- `code_search`：ripgrep 搜索（仅 workspace + 允许的 repo 范围）。

**运行（RunService 封装）**
- `run_create` / `run_cancel` / `run_status` / `run_logs` / `run_outputs`
- `run_retry` / `run_resume`（按引擎能力提供）

**工作流（WorkflowService + registry 封装）**
- `workflow_search` / `workflow_get` / `workflow_validate`
- `workflow_draft_create` / `workflow_draft_apply_edit` / `workflow_draft_diff`
- `workflow_publish_beta`（P0 只要求 beta 可用闭环）

**镜像（ImageService 封装）**
- `image_list` / `image_pull`（allowlist + policy）

### 7.2 风险分级与策略模式

每个工具/动作绑定风险级别：
- `read`：只读（扫描/读取/查询），默认自动执行。
- `act_low`：低风险写（生成 samplesheet、写临时文件、写 run inputs），SAFE_AUTO 下可自动。
- `act_high`：高风险副作用（启动 run、pull image、修改 workflow、发布 workflow），默认需要审批。

Session policy modes：
- `ASK_ALL`：除 read 外一律审批。
- `SAFE_AUTO`：read/act_low 自动；act_high 审批。
- `FULL_AUTO`：无审批（但仍有硬禁，例如 `.env`、`.git/`、越界路径）。

### 7.3 统一返回与记录

ToolRuntime 必须统一返回：
- `status`: success|error
- `data`: 结构化结果
- `error`: 可读错误信息 + 分类码（用于自愈）
- `artifacts`: 可选（文件路径、diff、preview）
- `meta`: duration_ms、tool_version、truncated 等

并且每次调用都会：
- 写 `agent_tool_calls`（DB index）
- 追加 `action_log.jsonl`（证据链）
- 通过 SSE 发 `agent.artifact`（以及可选 `agent.step.*`）

## 8. Approvals：把“危险动作”变成可点确认的卡片

审批对象（P0）：
- `file_diff`：对 workspace/workflow 的修改（显示 diff）。
- `command`：运行命令（如果仍存在 safe shell 类能力）。
- `run`：启动/重试/恢复 workflow run。
- `publish`：发布 workflow（beta）。
- `image_pull`：拉取镜像（含来源与 tag）。

审批生命周期：
1) Executor 运行到需要审批的 step -> 创建 `agent_approvals` -> SSE `agent.approval.requested`（或用 `agent.artifact` 包装）。
2) UI 调用 approve/reject API。
3) Executor 继续/终止，并在 ledger 记录决策与操作者。

## 9. Workflow 生命周期（P0：先闭环 beta）

P0 只做最小闭环：
- workflow 允许在 run dir 内“就地修复/生成”。
- 成功跑通后可 **publish 到 registry 的 beta latest**。

Stable/SemVer/回归矩阵/undo checkpoint 作为 P1+（但本 SPEC 为它预留数据结构与事件）。

## 10. Observability（P0：本地可调试 + 可选 LangSmith）

P0 必须：
- 本地 DB trace 继续可用（prompt/response/tool 的截断记录）。
- 每个 run 有 action_log + provenance，UI 能展示 tool calls 时间线。

可选增强：
- LangSmith tracing（dev 全量，prod 采样 + 脱敏/截断）。

## 11. 兼容性：保持现有 API/SSE 不崩

必须保持：
- `/api/v1/agent/message` 仍能工作（旧前端无感）。
- 现有 SSE 主事件：`agent.thinking` / `agent.plan` / `agent.artifact` / `agent.message` / `agent.done`。

允许新增（前端逐步接入）：
- `agent.run.*` / `agent.step.*` / `agent.approval.*`（结构化事件，更好做时间线 UI）。

## 12. 安全与边界（硬规则）

- 文件工具必须有 workspace root + path normalize，禁止越界；禁止 `.env`、`.git/` 等敏感路径。
- 任何执行类工具必须 allowlist 或明确“通过 RunService/WorkflowService/ImageService”进行（减少任意 shell）。
- 所有写操作尽可能落在 run dir 或 workflow draft dir，减少污染用户 workspace。

## 13. 成功标准（可衡量）

- GT1–GT7 全部通过（手工验收 + 最小自动化测试）。
- agent run 失败时，至少能给出“失败分类 + 下一步建议/修复提案 + 可重试动作”。
- UI 可以展示：计划、step 列表、tool calls、审批卡片、run 状态/日志入口。
