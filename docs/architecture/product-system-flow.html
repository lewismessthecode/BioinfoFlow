Bioinfoflow / Customer & Product View

从用户进入，到 agent 驱动执行，再到 scheduler 落地

这张图刻意不用工程内部视角去堆类名，而是把客户和产品真正关心的链路压成一张总览图： 用户怎么进入系统，workflow 怎么注册，run 怎么创建，镜像什么时候介入，agent 和平台如何协作， 最终又是怎么被 scheduler 接住并执行。

当前系统的主心骨 GET /workflows/{id}/form-spec + POST /runs 是 run 体验的唯一真契约。
Agent 不是另一套平台 agent 负责理解用户、组织步骤、申请审批，但真正的 workflow/run 仍应回到平台主链路。
Scheduler 是执行中枢 它接住已编译好的 run，做排队、资源判断、调度执行、状态回写和事件广播。

一张图看主链路

这条轨道是“标准产品流程”。用户可以从 UI、agent、demo 快速入口进入， 但真正开始执行前，都会尽量汇入同一个 run 编译与调度链路。

核心链路 可选入口 Agent 分支
1

登录与工作区

Identity / Workspace
  • 用户登录后，请求携带 Better Auth session。
  • 后端用 get_current_user 解析用户与 workspace_id。
  • 首次进入时更像“自动确保 membership + default workspace”，不是显式 workspace CRUD。
2

项目与入口

Project Scope
  • 用户进入某个 project，或通过 /projects/default 获得默认项目。
  • 从这里开始，文件、会话、run、terminal 都被 project scope 接住。
  • 这是产品层真正的“工作上下文”。
3

Workflow 注册

Workflow Catalog
  • 用 /workflows/validate 做源码预校验。
  • 用 /workflows 落库，抽取 schema / DAG / form spec。
  • 再用 /projects/{project_id}/workflows/{workflow_id}:bind 绑定到项目。
4

Run 准备与提交

Canonical Envelope
  • 前端或 agent 先取 /workflows/{id}/form-spec。
  • 用户填写 values，可带 options。
  • 统一走 POST /runs，而不是多套参数格式。
5

编译、调度、执行、回传

RunCompiler -> Scheduler
  • RunCompiler 做校验、路径解析、附件物化、launch snapshot。
  • run 入队后由 scheduler 看资源、排队并调用 execution backend / engine adapter。
  • 日志、状态、DAG、输出通过 API 与 SSE 回流到前端。

给客户和产品的解释方式

如果不想讲太多内部名词，可以直接按下面这三个层次去解释。

客户看到的是一个连续体验

  • 先进入自己的 workspace / project。
  • 选一个 workflow，系统给出表单。
  • 提交后系统自己去准备输入、执行环境、调度资源与追踪状态。
  • 如果用 agent，对客户来说只是“多了一个会帮我组织步骤的助手”。

产品真正要守住的边界

  • 一套 run 契约，不要前端、agent、脚本入口各自发明 payload。
  • 一套 project/workspace 授权，不要边缘入口比主链路更宽松。
  • agent 是体验增强层，不是平台核心的平行实现。
  • scheduler 是执行中台，负责把已经合法的 run 安全落地。

关于 “新建 workspace” 的现实口径

按当前代码，用户登录后并没有一条明确暴露给前端的 “workspace create API”。 更准确的说法是：系统在登录后自动确保用户属于某个 workspace，并确保默认 workspace 存在。 如果你要对外讲产品流程，建议写成 “进入或加入 workspace”，不要写成 “前端显式新建 workspace”。

关于 Bundled Workflows 的产品口径

现在这些 bundled workflow 目录应该被表述成“测试 / 本地验证素材”，而不是对外产品功能。 对客户和产品统一说法时，建议只讲标准 workflow 注册与 run 提交流程，不再讲 demo service。
